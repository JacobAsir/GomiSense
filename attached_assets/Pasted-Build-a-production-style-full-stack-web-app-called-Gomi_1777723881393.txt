Build a production-style full-stack web app called “GomiSense”.

High-level goal:
GomiSense is a Japan-first utility AI app that helps residents in Japan figure out how to sort and dispose of household waste correctly.
Users can upload an image of an item, speak the item name, or type the item name.
The app uses vision + voice + municipal rule lookup to return the correct disposal category and preparation guidance.

Important product principle:
This is a locality-aware waste sorting assistant, not a generic chatbot.
Municipal rules vary by city/ward, so answers must be based on a selected municipality profile and explicit rules.
When uncertain, the app should say so and give fallback search guidance.

This version should support:
- image upload
- optional voice input
- text search
- city/ward rule profiles
- multilingual result explanation
- stateless MVP by default
Do NOT add auth or a database unless explicitly asked later.

Primary user problem:
Waste sorting in Japan is strict, local, and often confusing.
Users need quick, practical guidance for:
- burnable / non-burnable
- PET bottles
- cans
- glass
- paper
- plastic packaging
- oversized waste
- hazardous or special disposal

Target users:
- Japanese residents
- foreign residents in Japan
- students and new movers
- families
- municipalities later as B2B potential

Municipality logic:
Build system to support rule packs per municipality.
Start with mock rule packs for 3 to 5 municipality profiles.
Each rule pack should include:
- category mapping
- synonym list
- collection notes
- special handling notes
- fallback instructions

Examples:
- PET bottle requires cap and label removal
- spray cans require emptying before disposal
- batteries need special collection
- ceramic cups may be non-burnable depending on municipality
- large items become oversized waste

Input pipeline:
Step 1: intake
- validate image/audio/text input
- require municipality selection
- accept one or more input types

Step 2: voice/text normalization
- transcribe voice if used
- normalize Japanese and English item names
- map synonyms

Step 3: image interpretation
- detect likely object/item class
- return multiple candidates with confidence

Step 4: rules engine
- resolve item against municipality rule pack
- use deterministic rules
- do not rely on LLM for final disposal category
- if uncertain, return top candidates and say “check municipality site / hotline”

Step 5: explanation layer
- use Groq for plain-language bilingual explanation if needed
- explanation must be grounded in the rule engine result
